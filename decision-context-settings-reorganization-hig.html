Settings Reorganization - Apple HIG Analysis

Four approaches aligned with Human Interface Guidelines and macOS design patterns

Settings Reorganization Strategy
Based on comprehensive analysis of Apple's Human Interface Guidelines, macOS System Settings patterns, and apps like Safari, Mail, and Photos. Each approach resolves the General tab bloat while following established Apple design principles.
Task-Based Organization (Recommended by HIG Analysis)
Overview

Organize settings by user workflow and what users are trying to accomplish. Five tabs follow recording-to-output progression: Recording (shortcuts, audio input, management), Transcription (models, quality), Output (AI enhancement, corrections, delivery), Automation (PowerMode), and General (app preferences). Matches patterns in Photos, Safari, and Mail.

Strengths
  • Strongest alignment with HIG clarity principle - each tab answers a specific user question
  • Matches Apple's task-based patterns in Photos (Library/For You/Albums) and Safari (General/Tabs/AutoFill)
  • Natural workflow progression from input to output matches user mental model
  • Groups all recording controls together - shortcuts, audio devices, and management in one location
  • Progressive disclosure works naturally - new users stay in Recording/Transcription, power users discover Output/Automation
  • Scales well for future features - new capabilities fit logically into existing workflow categories
Considerations
  • PowerMode isolated in dedicated Automation tab may feel sparse initially
  • Requires creating four new view files and redistributing General tab sections
  • Existing users need to relearn tab locations though structure is more intuitive
  • Five total tabs slightly increases tab bar width compared to four-tab approaches
Technical Details
Tab Structure
Recording (shortcuts, audio input/management), Transcription (models, quality, API keys), Output (AI enhancement, corrections, paste method), Automation (PowerMode), General (appearance, permissions, privacy, backup)
Implementation Scope
Create RecordingSettingsView, TranscriptionSettingsView, OutputSettingsView, AutomationSettingsView. Redistribute 9 General sections across new views. Update SettingsTab enum with 5 cases.
HIG Principles Applied
Clarity through focused categorization, progressive disclosure, visual hierarchy, consistency with macOS patterns, benefit-focused naming that describes user actions
User Impact
Most intuitive for new users. Reduces cognitive load by grouping related tasks. All settings preserved during migration. One-time notification about reorganization.
Best for creating Apple-like user experience with clear workflow progression. Ideal when prioritizing discoverability, reducing cognitive load, and establishing scalable structure for future features. Most aligned with HIG analysis.
Feature-Domain Organization
Overview

Group settings by technical feature domains aligned with system architecture. Four tabs organize by input/output stages: Input & Recording (shortcuts, audio, PowerMode), Transcription Models (model selection, quality, APIs), Output & Enhancement (AI transformation, corrections, paste method), and App Settings (preferences, permissions, privacy, backup). Follows macOS System Settings domain-based categorization.

Strengths
  • Fewest tabs (4 total) creates cleanest tab bar with clear visual hierarchy
  • PowerMode logically grouped with recording input rather than isolated
  • Matches System Settings pattern of domain categorization like Sound, Keyboard, Displays
  • Clear technical boundaries make it easier for developers to locate settings during debugging
  • Consolidates input concerns - recording triggers, audio devices, context awareness together
  • Power users who understand technical architecture can navigate efficiently
Considerations
  • Less intuitive for non-technical users who may not understand input/output distinctions
  • Output & Enhancement tab combines transformation with corrections which serve different purposes
  • Input & Recording tab becomes large with shortcuts, audio devices, management, and PowerMode
  • Workflow fragmentation - users still jump between Input and Output tabs for complete setup
Technical Details
Tab Structure
Input & Recording (shortcuts, mouse, audio devices/management, PowerMode), Transcription Models (models, quality, APIs), Output & Enhancement (AI, corrections, paste), App Settings (preferences, permissions, privacy, backup)
Implementation Scope
Create InputRecordingSettingsView (consolidate multiple sections), OutputEnhancementSettingsView (merge Transformation + corrections), AppSettingsView. Merge existing ModelManagementView. Update SettingsTab enum to 4 cases.
HIG Principles Applied
Domain-based categorization similar to System Settings, visual grouping of related technical features, clear separation of concerns, reduced tab count improves scannability
User Impact
Efficient for power users familiar with technical concepts. Larger tab sections require more scrolling. Best tab-to-content ratio balances organization with detail.
Best for maintaining architectural clarity and minimizing tab count. Ideal when targeting technically-savvy users who prefer domain-based organization and when keeping tab bar compact is priority.
Progressive Complexity Organization
Overview

Organize settings by complexity level and frequency of use. Five tabs progress from essential to advanced: Quick Start (recording shortcuts, basic audio, common preferences), Transcription (models, languages, quality), Enhancement (AI transformation, corrections, output), Automation (PowerMode, context rules), and Advanced (detailed audio controls, permissions, privacy, experimental features). Follows Xcode and Final Cut Pro progressive disclosure patterns.

Strengths
  • Optimized for new user onboarding - Quick Start contains only essential 20% of settings
  • Reduces overwhelm by hiding advanced features until users are ready to discover them
  • Clear complexity signals - users know Quick Start is safe, Advanced requires understanding
  • Follows progressive disclosure HIG principle - make essential information easy to find
  • Fastest path to basic functionality - new users never leave Quick Start initially
  • Matches professional app patterns like Xcode Behaviors or Final Cut Pro preferences
Considerations
  • Subjective complexity categorization requires assumptions about what is essential vs advanced
  • Power users must navigate to Advanced tab frequently which may feel like features are buried
  • Artificial separation of related features - audio controls split between Quick Start and Advanced
  • Complexity levels may not match all user mental models of feature importance
Technical Details
Tab Structure
Quick Start (shortcuts, basic audio, preferences), Transcription (models, languages, quality), Enhancement (AI, corrections, output methods), Automation (PowerMode, context rules), Advanced (detailed audio, permissions, privacy, experimental, backup)
Implementation Scope
Create QuickStartSettingsView (curated essentials), EnhancementSettingsView (expand current), AutomationSettingsView, AdvancedSettingsView (consolidate technical features). Significant redistribution of General sections.
HIG Principles Applied
Progressive disclosure at structural level, essential information given prominence, negative space and hierarchy guide users from simple to complex, matches pro app complexity gradients
User Impact
Dramatically improves new user experience. May frustrate power users who need frequent access to advanced features. Requires careful feature categorization to avoid confusion.
Best for optimizing new user onboarding and reducing interface complexity. Ideal when user research shows most users only need basic features and when following professional app patterns like Xcode or Final Cut Pro.
Minimal Reorganization (Conservative Approach)
Overview

Keep existing six-tab structure but redistribute General tab bloat. Split General into Recording (shortcuts, audio management, paste method) and Settings (appearance, permissions, privacy, backup). Merge Customisation into Enhancement. Rename tabs for clarity: Models to Transcription, Transformation to Enhancement. Minimal disruption while resolving primary bloat issue.

Strengths
  • Lowest implementation risk - smallest code change surface area reduces chance of bugs
  • Minimal user disruption - five existing tabs remain in same relative positions
  • Fastest to implement - only requires splitting GeneralSettingsView into two files and one merge
  • Easy rollback if users find new structure confusing - simple revert to previous state
  • Still addresses core bloat problem by dividing nine General sections into logical groups
  • Preserves all existing user settings and preferences without migration complexity
Considerations
  • Doesn't fully solve organization issues - related settings still scattered across tabs
  • Users still jump between Recording and Audio Input tabs for complete recording setup
  • Missed opportunity to establish clear organizational principle for future scalability
  • Tab naming remains somewhat generic - Recording and Settings don't clearly communicate scope
  • Doesn't leverage HIG patterns as effectively as more comprehensive reorganizations
Technical Details
Tab Structure
Recording (shortcuts, audio management, paste method), Transcription (renamed Models), Enhancement (Transformation + Customisation merged), Automation (PowerMode, experimental), Audio (existing), Settings (appearance, permissions, privacy, backup)
Implementation Scope
Split GeneralSettingsView into RecordingSettingsView and AppSettingsView. Merge DictionarySettingsView into EnhancementSettingsView. Update SettingsTab enum with renamed cases. Minimal structural changes.
HIG Principles Applied
Reduces cognitive load through section splitting, improves scannability by reducing per-tab content, maintains consistency with existing structure users understand
User Impact
Least disruptive to existing users. Incremental improvement rather than comprehensive solution. Zero settings migration needed. Can iterate further based on user feedback.
Best for addressing immediate bloat with minimal engineering effort and user disruption. Ideal when prioritizing low-risk incremental improvement over comprehensive reorganization, or when time constraints prevent larger refactoring.
